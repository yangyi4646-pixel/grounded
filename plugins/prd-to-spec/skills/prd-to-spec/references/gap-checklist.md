# Gap Checklist — PM 稿最常缺这些

> 用法:抽完骨架后逐类对照 source,缺/歧义就记成编号 gap(位置 + 缺什么 + 推荐默认 + 影响单元 + 优先级)。
> 标 ★ 的属**契约类**,阻塞所有单元,确认门里**先行单独过**。

## 总纲:两根正交的轴(查全 + 定值)
gap 不是平铺清单,是两根轴——**漏没漏靠完整性矩阵(查),怎么定靠契约轴(拍)**。下面 #0–#13 是这两轴的细化检查项。
矩阵按 `SKILL.md` 的复杂度路由启用;轻量档保留查漏意识,但不强制渲染全量网格。

### Stop-on-P0 + P1 截止点

gap 优先级决定流程停点:

| 级别 | 触发 | 行为 |
| --- | --- | --- |
| P0 硬确认 | 主对象、对象边界/基数、IA/入口路径、生命周期/状态机、权限/可见范围、id/存储、删除/迁移/引用完整性、版本/消费读取、字段语义归属 | 停止生成完整 SPEC。只写 `P0 硬确认请求 + gap 清单 + 占位骨架`;P0 解除前不得写 §0 具体契约值、单元详情、交互矩阵或验收断言。 |
| P1 条件确认 | 影响某单元实现、交互形态、默认值、错误恢复或运行策略,但不阻塞不相关骨架 | 可写 `推荐默认(待确认)` 继续草案,但必须写确认截止点和升级规则。到 `before handoff / before spec-to-code / before affected unit implementation` 未确认,升级为受影响单元 P0。 |
| P2 后置项 | 不影响当前规格冻结和实现路径 | 记录 Later / backlog,不阻塞下一步。 |

机械检查:有 P0 时运行 `scripts/check_stop_on_p0_confirmation.py <confirmation-or-output.md>`。含 P1/P2 gap 清单时运行 `scripts/check_confirmation_gap_levels.py <gaps.md>`。脚本通过只说明停机 / 截止点 / 升级规则结构合规,不说明业务判断正确。

### 轴一 · 完整性矩阵(按档启用,空格 = gap)
先判矩阵档位,再执行:

| 档位 | 矩阵要求 | 说明 |
| --- | --- | --- |
| XS | 不渲染矩阵 | 只写「无对象/权限/状态/存储变更」的影响判断 + 验收 |
| S | 局部矩阵 | 只覆盖本次触及的对象 + 操作;未触及操作可整行 `N/A` 并说明 |
| M/L | 完整矩阵 | 对每个对象逐格过;关联/连接对象必须独立成块 |

需要矩阵时,任一格空 → 要么补、要么显式标「本期不做 / N/A」,**不许默认略过**:

> 模板:复制 `templates/gap-matrix.md`。机械检查: `scripts/check_gap_matrix.py <gaps-or-spec.md>` 只判结构,不判业务对错。

| 操作 ＼ 关注点 | 权限 | 空态·默认态·加载态 | 错误·失败态·反馈态 | 状态·生命周期 | 数据一致性·引用完整性 | 消费·读取影响 |

| 增 create | | | | | | |
| 删 delete | | | | | | |
| 查 · 列表 | | | | | | |
| 查 · 搜索/筛选 | | | | | | |
| 查 · 详情 | | | | | | |
| 改 update | | | | | | |

> **列设计(2026-06-23:消费 / 数据一致性 升为关注点列;2026-06-28:loading/反馈态并入空态、错误态两列)**:旧版 4 列(权限/空态/错误态/状态流转)把 BI/配置对象高发的「改名同步、删除连带、孤儿值、写了谁读」这类**数据一致性 / 引用完整性 / 消费影响**风险逼得无处安放,只能硬塞进不对的列(场景集 3 门 A partial 的**列错配根因**)。新增两列各有归属:
> - **空态·默认态·加载态**:无数据 / 首次使用 / 搜索无结果文案与默认值,**外加该操作的加载/进行中态**(请求 in-flight 时显示什么、骨架屏 / loading / 禁用,加载何时结束)。原"感官反馈门"查的「在哪显示加载、状态有没有声明」并入此列——**状态声明缺失 = 此列空格 = gap**。
> - **错误·失败态·反馈态**:校验失败 / 越权 / 唯一冲突 / 网络超时 / 部分回滚的处理,**外加该操作成功/失败后给用户的反馈态**(toast / 提示 / inline 错误 / 二次确认结果反馈——成功提示、失败提示、提示文案与位置)。原"感官反馈门"查的「反馈态(toast/提示)有没有声明」并入此列——**反馈缺失 = 此列空格 = gap**。
> - **数据一致性·引用完整性**:改名是否同步到引用处、删除被引用对象的连带、关联对象孤儿值、跨视图同名字段口径一致。
> - **消费·读取影响**:每个操作「写了之后谁读到 / 在哪生效」——把"消费"从一个**操作行**改成一根**关注点列**(每个 CRUD 格都回答"改/删了谁读到"),比单留一行更能逼出消费缺口。
>
> **状态声明的唯一归属(2026-06-28 合并感官反馈门)**:loading/加载态、反馈态(toast/提示)这类**状态声明**统一由本矩阵的「空态·默认态·加载态」「错误·失败态·反馈态」两列负责——每个 CRUD 格逐格回答"加载时显示什么、成功/失败给什么反馈",空格 = 状态声明缺失 = gap。原独立的"感官反馈诊断门"已撤;其原本要查的状态声明并入这两列,**矩阵是这些状态的唯一归属**,不再另设门。
> 这轴大多能机器枚举,人审「有没有漏格」。**漏格 = 静默不完整**——"靠模型自己想 CRUD"会漏。矩阵是安全网,按风险展开:XS 不画全表,S 画局部,M/L 画完整。**完整性是 skill 结构,不是模型能力。**

#### 真渲染强制(产出格式硬规则,空挂标题 = 没装,2026-06-22)
当档位要求渲染矩阵时,`gaps.md` 里这一节**必须是真网格,不是顶着「完整性矩阵」标题的扁平 CRUD 列表**,否则视同完整性轴没做:
1. **逐对象独立成块**:本需求触及的**每个对象**(含**关联/连接对象**,如「指标↔标签关联」「目标↔指标绑定」)各起一张表头相同的网格——**禁止多对象塞进一张「功能点」清单**。漏列关联对象 = 漏格(关联对象的删除连带 / 谁读最常被吞)。
2. **逐格非空**:每格要么填该对象在该(操作×关注点)下的具体结论,要么写显式 `N/A` + 一句理由(如「标签无状态机 → 状态·生命周期 N/A」)。**空格 = 未决 gap**,不许留白、也不许整列整行省略。**全篇至少要有一处显式 `N/A`**——一处 N/A 都没有,通常说明在偷偷略过不适用格而非真逐格过。
3. **三经典漏格各有专列(改列后不再无处放)**:**删除连带** → 「删 delete × 数据一致性·引用完整性」格;**改名同步** → 「改 update × 数据一致性·引用完整性」格;**消费谁读** → 各操作的「消费·读取影响」列。**不靠正文叙述兜、也不再硬塞进状态 / 错误列。**
4. **一格一关注点(防伪网格·格内信息混装)**:每格只写**该(操作×关注点)交叉点的一条结论**;**禁止把多个独立 gap 或多条 recon 结论压进同一格**。一格涉及多个未决项 → **只挂 gap 编号**(`⚠️ G6`),详情留在 gap 清单。**注:改名同步 / 删除连带 / 谁读现在各有专列(数据一致性列、消费列),不再是"无列可放硬塞"的来源**——场景集 3 门 A partial 的列错配根因已由改列消除;若仍出现一格塞 ≥2 个独立 gap 编号 / ≥2 条 recon 结论 = 信息混装,拆。**网格骨架对(不漏格)但格内混装 = 仍是伪网格,只兜了下限。**
> ❌ 反模式(场景集 3 实拍):标题写「完整性矩阵」,底下却是 `打标 / 按标签筛选 / 增删改` 三句扁平 CRUD,无网格、无一处 N/A、第二对象「指标↔标签关联」未独立成行——**三漏格靠模型直觉恰好想到、网格退化成列表 = 安全网根本没装**。
> 触发与豁免(保持 scoped):XS 可不画表但必须写明为什么不触及对象生命周期;单对象、纯展示类薄需求可整张表标 N/A 跳过,但**得显式标**,不得静默不画表。

### Grep Evidence 模板(门 D)

当写出「系统无 / 未命中 / 没有现成能力 / no match」这类结论时,必须配 `templates/grep-evidence-block.md`。机械检查: `scripts/check_grep_evidence.py <gaps-or-spec.md>` 只检查 Wordset / Command / Hits / Conclusion 是否存在,不判断搜索词是否足够。

### Approval / Lifecycle Gate(审批/生命周期操作门)

当需求触及审批、发布/上线、下线、删除、撤回、重新提交、回调、第三方/外部处理、配置开关、版本生效或消费读取时,必须复制 `templates/approval-lifecycle-gate.md`。机械检查: `scripts/check_approval_lifecycle_gate.py <gaps-or-spec.md>`。

这不是项目规则,是防静默裁剪机制:删除/批量/pending 可编辑性/重复提交/回调/第三方只读/配置面/消费读取都必须被确认、标 gap 或写 `N/A + reason`。模型不得以"V0 好控"为由自动排除风控对象。

**N/A 收紧规则**:审批/生命周期门里,`输入未提 / PRD 未写 / 待确认 / 需后续确认 / gap / 阻塞` 都不是 N/A 理由,而是 gap 理由。真正的 `N/A / 不适用` 必须说明“本需求明确不触发该风险”,并给来源;否则机械检查失败。

### 状态声明(loading / 反馈态)归属:完整性矩阵

loading/加载态与反馈态(toast/提示)这类**状态声明**没有独立的门,统一由上文「轴一 · 完整性矩阵」的「空态·默认态·加载态」「错误·失败态·反馈态」两列负责——每个 CRUD 格逐格回答"加载时显示什么、成功/失败给什么反馈",空格 = 状态声明缺失 = gap。矩阵是这些状态的唯一归属。

> 注:用户给「不好看 / 很怪 / 更丑 / 感觉不对 / 信息重复 / 没有区分 / 对不上 / 展开收起关系不对」这类感官/症状反馈,且症状指向**主对象不清 / 对象边界 / 字段语义 / 信息归属 / 信息重复**时,走下文「Information Ownership Matrix(信息归属矩阵)」做结构诊断;不得先改卡片、折叠、容器、边框、内边距或样式。

### Information Ownership Matrix(信息归属矩阵)

当需求涉及 B 端 BI / 指标 / 目标 / 数据集 / 配置页中的多对象关系、字段映射、上游/下游消费、头部摘要、展开/收起、卡片/表格/详情结构时,必须复制 `templates/information-ownership-matrix.md`。机械检查: `scripts/check_information_ownership_matrix.py <gaps-or-spec.md>`。

5 项 canonical 定义(主对象 / 信息归属矩阵 / 头部摘要职责 / 展开区职责 / 不重复展示规则)见 `references/information-ownership.md`(单一权威源,改定义只改那里)。

### 轴二 · 契约轴(填进格子的"值",人在确认门拍)
格子要落地得先定**契约值**——机器没上下文/权威,猜就漂,**必须人拍**(下方 ★ 维度即此轴):范围边界 / 对象基数 / 字段作用域(共享 vs 各配)/ 枚举口径 + 跨层一致 / 状态机 / 交互形态 / id · 存储规则。
> 一句话:**完整性轴查"覆盖全没",契约轴拍"每个怎么定"。** 二者都过,gap 才算清。

## 0. ★★ 产品/使用价值

> 这是**需求补全的第一层**,不是拒绝入口。逐条问,缺了别替用户拍,但也别把补全责任原样退回用户。

- 为什么做?成功指标 / 用户的活(JTBD)是什么?
- 产物**在哪被使用/消费**?用时什么**效果/体验**?(不只是"能配置",是"配了之后在哪生效、用户在哪感知价值")
- **管理侧 vs 消费侧谁是重点**?(别只造管理 CRUD 而漏了价值所在的消费侧)
- 使用旅程:从产生需求到拿到价值,走哪几步?
- **不做会怎样**?(验证是不是真需求)

⚠️ PRD 没覆盖以上 → 作为**产品缺口**进入补全确认门:先给候选场景/默认路径/影响后果,再让用户确认、改写或暂缓。别把只覆盖管理 CRUD 的薄 PRD 冻结成 SPEC,也别只说「请补充完整需求」。

> **PASS 判定(防 borderline 漂移)**:PRD 须明示 (a) **消费场景/在哪用** + (b) **使用效果或成功指标**——缺任一即 ★★ blocking,但处理方式是补全确认,不是退回;若 (a)(b) 已有、仅缺「不做会怎样 / 管理vs消费谁重点」,则降为 **P2 一句话过**(明显=有默认,不是跳过理由)。

> **拆问阶梯**:用户答不上「为什么做」时,拆成:谁会主动来用?他现在在哪一步卡住?不做会继续用什么 workaround?这个功能成功后,他少做哪一步/少等谁/少出什么错?这些答案足以反推价值与验收。

## 1. ★ 对象模型与关系
> 实体若是**已有对象**(非本需求新建),先 recon 其真实形态(类型 / 模型定义)再判字段与基数——别按 PRD 举例推断(PRD 举例常把多维对象写成单值)。
- 核心实体有哪些?字段、类型、必填性?
- 实体间关系基数:一对一 / 一对多 / 多对多?(基数判错是返工高发点)
- 谁是"行/记录"的载体?嵌套还是平铺?

## 2. ★ 枚举 / 状态 / 状态流转
- 有哪些枚举值?完整集合(含"其它/未设置")?
- 状态机:合法状态有哪些?谁能流转到谁?有没有草稿/发布?(确认"无草稿直接上线"这类约束)
- 默认状态、终态、不可逆流转?

## 3. ★ id 规则 / 存储键 / 共享契约
- 行 id 怎么生成(后端给 / 前端 uuid / 复合键)?稳定吗?
- localStorage / 配置 / URL 参数的存储键名与结构?跨单元共用的键由谁定义?
- 多个单元共享的对象模型/类型定义放哪、谁拥有?(漂移高发,必须 §0 冻结)

## 4. 范围 / 价值 / 明确不做 / 影响面
- 解决什么问题、为谁?成功指标?
- **out-of-scope**:本期明确不做什么?后端是否 de-scope?
- **Affected Surfaces**:本轮直接改变哪些页面、入口、状态、消费面?是当前页面、列表、详情、配置侧、消费侧、导出/订阅/通知中的哪几个?
- **Non-Goals / 不改面**:相邻但本轮不改的面必须写实,例如兄弟页、列表/详情另一端、批量入口、导出、权限配置、下游消费展示。
- **Propagation Rule**:是否因共用对象、常量、枚举、语义资产、字段渲染器需要连带同步?默认不扩散;若要同步,写同步面、理由和确认来源。
- **局部反馈边界**:来自当前截图、当前 URL、当前页面的反馈默认只冻结 current surface;若用户说「所有」「统一」「各处」「同对象全链路」才扩散,否则作为确认项问。
- 谁的会话/负责人拥有哪个单元(是否拆成多个独立会话)?

## 5. 单元拆分与依赖
- 能拆成哪些"有终点的小单元"?
- 单元间依赖顺序?谁共享 §0 契约?改一处会牵动谁?
- 哪些可并行、哪些必须串行?

## 6. 数据字段口径
- 字段来源 / 类型 / 单位 / 精度 / 时区 / 本地化?
- 默认值、必填 vs 选填、最大长度?
- 同名字段在不同视图口径是否一致?

## 7. 业务规则与计算
- 计算公式、聚合方式、优先级?
- 触发条件、生效时机?
- 共享配置是"每组一份"还是"全局一份置顶"?(作用域判错易返工)

## 8. 边界 / 异常 / 空态
- 空态:无数据 / 首次使用 / 搜索无结果 文案?
- 超长 / 超量 / 上限 / 分页?
- 0 / 负数 / 极值 / 小数精度?
- 无权限 / 越权?重名 / 重复提交 / 唯一冲突?
- 并发(同时编辑、最后写入 vs 锁)?网络失败 / 超时 / 部分回滚?
- 关联对象被删后,依赖它的入口/记录怎么处理?

## 9. 权限与角色
- 哪些角色可见 / 可改 / 只读?
- 未登录 / 无权限的行为(入口禁用还是隐藏 + 文案)?
- 谁接收输出 / 通知?

## 10. 实现路径(落到代码)
- 落在哪些目录 / 组件 / model 层?
- 复用哪些现成模块(优先 `src/core`、`src/model`)?
- 新增类型 / 接口放哪?

## 11. 交互细节
- 下拉 vs 输入框 vs checkbox?单选 vs 多选?(交互类型易判错)
- 控件**位置**(共享控件置顶还是内嵌、入口放哪——位置错也是返工源)?
- 必填 vs 选填、默认值、占位符?
- 确认 / 取消按钮?可撤销?二次确认?
- 进入即空白 / 跳转目标对不对(编辑入口跳到哪)?

## 12. 可测验收
- 每条能写成 Given-When-Then 吗?写不出预期结果 = 信息缺失。
- 一个 story ≤ 3 条验收;超了说明 story 太大该拆。
- 验收是否绑死了 URL/控件实现(应描述行为,不绑实现)?
- "做好为止""友好""快"这类模糊词是否已量化?
- 验收写成**可勾验断言表**(编号 + GWT + 检查方式 + 期望),见 `references/acceptance-as-eval.md`;失败断言 = gap 回流锚点。
