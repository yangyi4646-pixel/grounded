# 确认门(Clarify Loop)做法

确认门是**硬卡点**:写 SPEC 实体内容前必须过。目标——用最少来回把 gap 清空,绝不臆断。

薄输入时,确认门也是**需求补全门**:agent 先展开候选 PRD 骨架和 gap 树,再让用户确认/纠偏。用户不清楚时继续拆小问题,直到问题落到用户能判断的业务事实或后果。

## 一、把 gap 整理成可批量确认的清单
每条 gap 一行,固定字段:

```
#<id> | [P0/P1/P2] | [契约★ / 单元n] | <位置:哪份源/哪节> | <缺什么 或 哪里歧义>
       推荐默认:<具体默认 + 一句理由>
       影响:<阻塞哪个单元>
       确认截止点:<P0=现在停 / P1=before handoff|before spec-to-code|before affected unit implementation / P2=Later>
       升级规则:<P1 到截止点未确认时,升级为哪个受影响单元的 P0>
```

规则:
- **每条必带推荐默认**——让用户只需 确认/改/暂缓,而不是从零回答。
- 默认要**具体可执行**(「add 按钮」而非「某种交互」)。
- 写得出默认 = 大概率确认即过;写不出 = 真盲点,标记原因。
- 薄输入的默认必须标来源:「领域惯例推断 / 现有系统 recon / 同类功能对标 / 用户已确认」。不得把推断默认伪装成已决事实。
- **P0 / P1 / P2 是流程控制,不是严重程度标签**:
  - **P0**:主对象、对象边界/基数、IA/入口路径、生命周期/状态机、权限/可见范围、id/存储、删除/迁移/引用完整性、版本/消费读取、字段语义归属。命中即停,只交付 `P0 硬确认请求 + gap 清单 + 占位骨架`,不得继续写完整 SPEC。
  - **P1**:影响单元实现或交互选择,但不阻塞不相关骨架。可以写 `推荐默认(待确认)` 继续草案,但必须写确认截止点和升级规则。到截止点未确认,升级为受影响单元 P0。
  - **P2**:后置增强或可逆低风险项,记录 Later,不阻塞下一步。
- 清单本身常 >80 行 → **落盘**到 SPEC 同目录(`docs/specs/<需求>/notes/gaps.md`),对话只引用编号。

## 二、用 AskUserQuestion 分组批量问(契约先行)
先按需求档位决定确认门重量:

| 档位 | 确认门做法 |
| --- | --- |
| XS | 无契约/权限/状态/存储变更时,不单独开确认门;在摘要里写「无待确认」+ 1-3 条验收即可 |
| S | 最多 3 个确认项;每项必须带推荐默认 + 影响;超出 3 个或触发契约类 gap 即升级 M |
| M/L | 使用下方完整分组批量确认 |

轻确认不是免确认:只要问题会改变对象模型、作用域、状态、id/存储、跨单元契约,必须升级并走契约组。

- **第一组只放 P0 契约★**(主对象/对象边界/对象模型/行 id/枚举状态/存储键/权限/生命周期/消费读取/迁移引用)。它阻塞全部单元或关键单元,必须先冻结。出现 P0 时,不得继续生成完整 SPEC;复制 `templates/p0-confirmation-request.md`。
- 后续组按单元归类(单元2 一组、单元3 一组),每组 ≤ 4 题。
- 每题 = 问题 + 2~4 个选项,其中一个标「推荐(默认)」,并留「我来改」「暂缓」。
- 一次发一组,不要一题一题来回。

提问措辞:
- 开放式逼出上下文:「共享下拉是全局置顶一份,还是每组各一份?」优于「下拉对吗?」
- 先陈述假设再请纠偏:「我假设并发以最后写入为准,如不同请指出。」

### 用户答不上来:降阶拆问
- **不要追问大词**:「请补充业务价值 / 请补完整 PRD」无效。
- **拆到可观察事实**:角色(谁用) → 场景(在哪一步用) → 触发(什么时候需要) → 后果(不用会怎样) → 频率/风险(高频还是低频、错了成本多大) → 权限(谁能看/改/审批)。
- **每次只解决一个不确定性**:对象模型、权限、生命周期、消费场景不要混成一个大问题。
- **给推荐默认但保留纠偏口**:「如果你不确定,我建议先按 X,因为 Y;风险是 Z。确认 / 改成... / 暂缓。」
- **仍不能判断时**:P0 / 高风险契约类标 blocking 并停在确认门;P1 可用推荐默认继续草案,但必须写确认截止点和回滚条件;低风险/可逆 UI 文案类可降为 P2 或 Later。**P1 不是默认生效**:到 `before handoff / before spec-to-code / before affected unit implementation` 仍未确认,自动升级为受影响单元 P0。

### 输出分区与 self-scrub(交付物级 C2 门)

确认门输出必须分成两段:

```
【给用户】
只写业务语义事实、业务后果、推荐默认、确认/改/暂缓项。

【内部留痕】
写 recon 证据、路径、符号、接口、枚举、行号、未命中的 grep 结论。
```

**给用户段禁用 token 类**:
- 路径 / 文件扩展名:`packages/`、`src/`、`.ts`、`.tsx`、`.js`、`.scss`
- 代码符号:`camelCase`、`snake_case`、`SCREAMING_SNAKE_CASE`
- 点号符号 / API / 枚举:`PG.FAV`、`INFO_MAP`、`/api/...`
- 实现缩写:`API`、`ID`、`URL`(除非用户需求本身就在定义这些业务字段)
- 行号 / 流程元话术:`line 12`、`L12`、`§0`、`AskUserQuestion`、`降级 rule`

**定稿前 self-scrub**:
1. 只扫 `【给用户】` 段。
2. 命中禁用 token → 把该 token 和相关证据句移到 `【内部留痕】`。
3. 用业务语言补回给用户可判断的信息,例如「现有详情页已经按分组展示字段」替代内部名。
4. 允许短业务缩写仅限明确业务域词(如 BI);不要默认放行 API、ID、URL,它们常把实现负担甩给用户。
5. 给用户段仍命中禁用 token 时,不得交付。

### 系统依赖类问题:先 recon、再业务化(5 道一级门)

> **触发**:默认依赖现有系统能力的 gap(权限门 / 组件复用 / 删除语义 / 通知入口 / 图表能力 / 设计系统控件)。**动作**:先 lightweight recon(grep 类型 / hook / 组件 / 路由),拿到真实符号,再下默认。下面 5 道门必过——**recon 是出问题前的必做前置;符号留痕在内、业务语义对外**。
> (5 门一句话:**门 1 有据** · **门 2 分层** · **门 3 姿态** · **门 4 不替用户拍** · **门 5 交付前自检**。)

#### 门 1 · 证据门(recon 出的每个「有 / 无 / 可复用」主张都要有据)
- **现有实现证据分级(防 exemplar 伪造)**:任何「已有用例 / 可对标实现 / 可复用模式」主张都必须先分档,并在内部留痕带可点路径;**库存在 ≠ 有现成用例,类似能力 ≠ 可直接复用**。

| 证据档 | 允许说法 | 禁止说法 |
| --- | --- | --- |
| 精确命中 | 已验证同对象/同场景已有用例,内部留路径与能力边界 | 无路径就说「已有」 |
| 类似能力 | 有相似机制,但对象/生命周期/权限不同,只能作为参考 | 说成「可直接复用」 |
| 仅库存在 | 技术库或基础组件存在,未命中业务用例 | 说成「已有拖拽/收藏/审批等实现」 |
| 未命中 | 未查到,列为实施风险或待工程确认 | 编一个 exemplar 补足论证 |

- **命中现有实现 → 列能力边界**:只有「精确命中」或「类似能力」可以作为默认依据;必须显式列其能力边界(端点 / 字段 / 对象差异 / 生命周期差异)。查无精确命中时,写「库存在、无现成用例」或「类似能力、对象不同」,不得升级措辞。
- **精确命中前必做「能力核验」(存在 ≠ 会干那件事,命门)**:把某符号判为「精确命中 / 可直接复用」前,**必须读它的实现、确认它真能干 SPEC 要的那件事**,不能只凭文件/符号/数组名存在就升档。例:判「列表批量打标可复用 `MetricBatchOperate`」前要读它的 `batchActions` 数组**有没有 set-tag 这一项**;若只有 `BATCH_AUTHORITY/MOVE/DELETE/EXPORT/OFFLINE`、无批量打标 → 它是「类似能力(有批量框架、缺打标动作)」**不是**精确命中,降级。证据:场景集 3 把 `MetricBatchOperate.tsx` 标精确命中、实为无批量打标,下游照搬会以为白嫖到不存在的功能。**能力核验结论入内部留痕**,格式:`精确命中? 符号X | 读实现:X 的 batchActions/枚举/props = [...] | 含目标动作? 是/否 | 结论:精确命中 / 降为类似能力`。**只渲染四档标签、不附「读实现」证据行 = 能力核验没做**。
- **「系统无 X」充分性门(防有方向感的虚构 grounding)**:下「系统无 X 能力 / 未命中」结论前,**内部留痕必须先列出已 grep 的关键词集 + 各自命中行号(或 ZERO_MATCH)**,且关键词集要覆盖该能力的**领域同义词全集**——判「无变更历史」要搜 `version|history|snapshot|audit|变更|快照|版本`,不能只搜一个 `history` 就下结论。词集不全于同义词全集 → 只能标 **「recon 未尽·待复核」**,**不得**写「未命中 / 系统无 X」。证据:Phase 1 P2-A 只看一个词就硬断言「全仓无变更历史」,漏掉摆在那的 `MetricVersionHistory.tsx`。格式:`无 X? | 词集:{a,b,c} | 命中:a@L12, b=ZERO, c=ZERO | 结论:未命中 / recon 未尽`。

#### 门 2 · 分层门(给用户段干净 = 符号内部化;与上方 self-scrub 同构)
- **符号内部化是交付物级,不只问句级(命门)**:符号 / 路径 / 行号 + **流程元话术**(『AskUserQuestion 不可用』『§0 占位』『降级 rule』)**全部收进 `【内部留痕】`**;**`【给用户】` 整段零符号零元话术**——非工程角色消费的是整份文档,问句干净但正文符号墙照样卡死他(角色越远离工程越致命:设计师 > 运营 > 产品 > 交付)。能用截图 / 界面指代就别用内部名(『点进指标后列着数据源/口径/责任人的那块卡』,而非 `INFO_MAP`)。

#### 门 3 · 姿态门(告知非反问 + 业务化问法)
- **问法模板**:`已核对:<业务语义事实> → 业务后果/建议 <X> → 请确认`(✅『查看与管理是两级权限,个人预警建议走查看权』 ❌『有 useMCReadPK 即可配置?』)。
- **告知 vs 反问(句式硬化)**:recon 已能定死的现状事实(如『变更记录现状只有管理者可见』『单父目录树挂不了多分类』),用户段**必须**走固定句式:**「已核对现状:X → 业务后果:Y → 推荐默认:沿用 X → 请确认:是否要扩展(如放给更多人 / 改一处)?」**。**禁止**把已知系统约束重抛成开放题——❌『变更记录给谁看?所有人还是仅管理者?』(现状已定=仅管理者可见)→ ✅『已核对:变更记录现状只有管理者能看;建议沿用,要不要扩展到有查看权的人?』。只有 recon 也定不了的,才作为真开放问题。
  - **交付前强制自检(C3 门,与 C2 self-scrub 同构)**:逐条扫 `【给用户】` 的每个问题 → 它问的事实是否在 `【内部留痕】` 已有 recon 结论?**是 = 违规开放题**,必须改写成上面的告知句式后才能交付。**「我内部知道现状、外面却开放问一下」= C3 失败**,不是可接受写法。把每个用户问题标注成 `[现状已定→告知确认]` 或 `[真开放→提问]`;凡内部 recon 已定死却标成"真开放"的,即违规。
- **反例(禁)**:把工程量词(hook 名、分隔符、组件名、行号)直接抛给用户判。

#### 门 4 · 不替用户拍门(recon 定不了的产品/契约判断 → 单列确认项浮给用户,带默认但不静默)
- **PRD 明确入口 / IA / 路径时的优先级门**:若 PRD 已写明页面入口、导航层级、对象归属或用户路径,这些是产品事实;recon 出来的现有容器 / tab / 菜单只能作为「实现落点候选」和成本依据。任何把 PRD 明写入口改成别的层级(如独立入口 → 某已有 tab)的建议,必须单列确认项:「PRD 原路径 X;现有代码可低成本落在 Y;推荐默认仍按 X。是否接受改为 Y?」未确认前,SPEC / §0 / 页面契约一律按 X 写。
- **局部反馈默认 current surface(防误扩散)**:用户基于当前截图 / 当前 URL / 当前页面提出的修改,默认只影响该 surface。若 agent 认为同对象的列表、详情、配置页、消费页、导出、通知或批量入口也应同步,必须在 Scope Boundary 的 `Propagation Rule` 单列确认:「当前反馈只改 X;若要同步到 Y/Z,会影响 A/B,建议默认不扩散 / 或建议同步到 Y,确认?」。未确认前不得把局部问题顺手推广到兄弟页或下游消费侧。
- **作用域轴常驻(凡需求含共享/私有/按项目歧义,必单列,不许默认隐式继承)**:需求一旦触及「谁能看到 / 用到这个对象」(共享 vs 私有 vs 按项目/团队/owner),**作用域必须单列成一条独立 gap**——**禁止**用默认让新对象悄悄继承所依附对象的作用域。该条固定四件套:① **recon 读到的语义依据**(所依附机制的端点有无 `owner`/`scope`/`teamId` 字段、是 admin 全局配置还是按 owner 私有)② **推荐默认 + 其来源档位**(精确命中 / 类似能力 / 仅库存在)③ 该默认的**业务后果**(谁会看到 / 改到)④ **请人确认是否就是这个作用域**。把「grep 到某机制存在」直接当「作用域 = 该机制的作用域」= 存在性→语义滑移(SKILL.md 原则#2 的 caveat),正是这条要防的。证据:场景集 3 把「标签挂在业务属性体系上(grep 命中多选业务属性)」悄悄当成「标签 = 业务属性的 admin 全局作用域」,没把「端点无 owner/scope 字段 = 全局」这个语义读出来单列告知,用默认让作用域隐式继承。**保持 scoped:需求无作用域歧义时本条不触发,不强行加问。**
- **作用域/分组轴的『页面形态』分类(M/L 新页触发,与可见性正交)**:当分组/作用域轴(主题 / 目录 / 团队 / 组织 / 空间 / 项目)落在**已有页面 IA 的资源域**上,除了问「谁能看到」,**必须再判它在页面上是哪种形态**——从 {① 左侧树 / 页面上下文 ② 顶部切换 Tab ③ 面包屑路径 ④ 表格筛选项 ⑤ 纯表单字段} **选一**,**禁止默认当成「筛选项 / 字段」**。判据:该轴是不是**路由状态**(进 URL、决定左树 / 列表 / 面包屑)而非过滤条件——是 = 页面结构轴(①③),§0 须冻结成「复用 X 域的『左树 + 面包屑 + 右列表』骨架」,不能写成一个筛选字段。未定 → 单列 gap 带推荐默认请确认。证据(场景集 7):指标标签管理把「主题 / 目录 / 团队」当筛选字段,漏了指标管理页是「主题 + 左树 + 面包屑 + 右列表」(`Metrics`/`BusinessMetrics`/`Hierarchy` 同骨架、仓库通用 IA),下游自造 BasicTable 独立页、与现有 IA 完全脱节。这是离散分类(5 选 1),prose 锁得住。**§0 冻结硬门(防「识别≠冻结」,场景集 7 RED→GREEN 残留)**:形态判成页面结构轴后,**必须在 §0 写实一条『页面 IA 复用契约』**(如『页面 IA:复用指标管理页骨架 = 主题 + 左树 + 面包屑 + 右列表』),**禁止**回落成 §0 里的一个『筛选字段』。判据:§0 里这条仍写成『主题筛选 / 目录筛选』而非『复用 X 域骨架』= 形态门没过、§0 不算冻结。**recon 里认出 IA 不算数,要落到 §0 契约行。**
- **「近乎能干」≠「就该复用」——近匹配 → 产品并列选项门(recon 命中高重合现有体系时触发)**:当 recon 发现**现有体系功能上几乎覆盖整条需求**(如『业务属性』之于『标签』:CRUD + 取值 + 筛选全有),『**复用现有 X vs 独立新建 Y**』是**产品判断、recon 定不了**——差异在**产品 / UX 语义**(受控词表 vs 自由标签、全局 admin 维护 vs 用户可现场建、扁平 vs 可分层),代码读不出来。此时确认门**必须把两条摆成同等清晰的并列选项**(各讲清后果),**禁止默认偏向「复用」、也禁止把「独立」弱化成顺带一句**;**推荐默认在这里保持中立**(或并列给两条取舍),让用户**主动选方向**,而非顺着默认滑进复用。判据:近匹配存在却只把「复用」当默认、没把「独立」列成同等选项 = 没过。证据(场景集 8 跨引擎实测):薄需求『给指标加标签』两引擎都把推荐默认设成『复用业务属性』,真实用户**否决、选了『独立自由标签』**;Codex 还把『独立』选项弱化掉——用户若照默认点确认,直接走错方向。这是离散 presence 判定(确认门有没有摆出两条同等选项),prose 锁得住。

#### 门 5 · 交付前防静默默认自检(meta:门 4 各项交付前逐一过,确保都浮成确认项)
- **交付前 · 防静默默认门(只管本次触发的弱门,没触发跳过)**:跨引擎实测(场景集 4)证实 **完整覆盖(矩阵 N/A)/ 穷尽核验(「无 X」断言)/ 作用域 / PRD 明确入口被源码落点覆盖** 这几处模型最不稳、最易**静默默认**。交付前对**本次实际触发的**门逐一过,核心动作是 **「不默认、交还用户」**(不是 AI 自己拍死):① 矩阵里拿不准是否 N/A 的格 → 标「待确认」进 gap,**禁静默填 N/A 或整格略过**;②「系统无 X」未按 grep 充分性门查全的 → 降成「已查 {词集} 未命中、建议工程复核」的**确认项**,**禁当既成事实写死**;③ 作用域有共享/私有/按项目歧义 → 带推荐默认 + recon 依据**浮给用户确认**,**禁默认隐式继承**;④ PRD 明确入口/路径而源码有更省事实作落点 → 默认按 PRD,把源码落点作为**是否调整入口**确认项,**禁直接改写 PRD 路径**。**边界(防确认门膨胀)**:这些门一律以**「带推荐默认的确认项(确认 / 改 / 暂缓)」**浮出,**不得**甩成开放问题;且仅触发的门生效,频率可控,**没触发的门一律不加**。本质:AI 最不可靠的这些处,老实说「我拿不准、你来定」,而非装作搞定 —— 与「用户暂缓的绝不偷偷默认」红线同源。

## 三、回填
- 确认 → 写进对应单元/§0,并在**决策日志**记 决策 + 出处 + 影响。
- 改 → 按用户答案写,同样记日志。
- 暂缓 → 进「⚠️ 待确认」表,标阻塞单元;该单元在解除前**不得动手**。
- 冲突解决(多源对账出的)也必须落决策日志,注明取了哪源、为什么。

## 四、更新场景(PM 改稿)做增量 diff
1. 新稿丢进 `source/`,**先 diff 旧稿**:新增 / 修改 / 删除 了什么?
2. 只对**变化点**重跑 gap 扫描,别全量重来。
3. 变化若动到 §0 契约 → 标"契约变更",走第一组确认,并通知所有受影响单元 + 更新 handoff。
4. 更新「源对齐声明」:蒸馏自新版本号 + 本轮改了哪些单元。
5. 旧决策被新稿推翻 → 决策日志追加一行(不删旧行,标"被 〈date〉 覆盖"),保留可追溯。

## 防漂移红线
- 任何在聊天里达成的确认,**当轮就回写 SPEC**,不靠记忆。
- 决策必带出处(PRD / 纪要 / 用户确认 + 日期),便于回溯"为什么这么定"。
- 用户暂缓的项,绝不"先按默认偷偷做"——它在 ⚠️ 表里,阻塞对应单元。
