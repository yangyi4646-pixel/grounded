# 验收即 eval(可勾验断言)

把 SPEC 的验收写成**可勾验断言**,而不是散文。目的:让「做完」客观(可检查的通过/失败),并给 gap 回流一个**锚点**——这是整条链路 evaluator-optimizer 闭环的地基。

## 断言表(每单元的「验收」用它)

| # | Given-When-Then | 检查方式 | 期望 | 通过 |
| --- | --- | --- | --- | --- |
| A1 | Given 前置 When 操作 Then 可观察结果 | 实拍 / UT / 命令 | 期望 observable 或值 | ☐ |

每条断言 = 一个**原子、可独立判定**的检查。

## 三种检查方式(按单元性质选)
- **实拍**(UI 单元):observable 用「出现 / 不出现 / 变成 X」描述,项目 dev 地址(如 localhost:3001)或 Figma 肉眼可判。
- **UT**(纯逻辑 / Model 单元):写/跑单元测试,断言函数输入→输出。
- **命令**(可复现):给一条命令 + 期望输出片段,跑了贴输出。

## 写好断言的规则
- **原子、互不依赖**:别「接上一条」,否则失败难定位。
- **一个单元 ≤ 几条核心断言**;超了说明单元太大,拆。
- **描述行为,不绑实现**:写「未选数据集时保存按钮禁用」,不写「`<button disabled>`」。
- **量化模糊词**:「快/友好/合理」→ 具体阈值或具体 observable;写不出期望值 = 信息缺失,转成 gap。
- **覆盖边界**:空态 / 超限 / 无权限 / 并发各至少一条断言。

## 完成判据
**全部断言「可检且通过」** + lint/UT 过 + 文案/埋点到位。
任一断言「未检」(当前 actor 无法跑)→ 标 `未检`,**不算通过**;不静默判过。

## gap 回流锚点(闭环关键)
- 实拍/校验发现差异 → 记成「**A# 失败 + 现象**」,而非长描述或截图原文。
- 回流为 SPEC 的**增量断言**(新 A#),带出处(实拍日期/谁发现),写进决策日志。
- 「失败的断言编号」就是 evaluator-optimizer 循环的客观输入——重生成后只需复跑这些断言。

## 与下游生成器的接口
- **spec→code**:断言里 `检查方式=UT/命令` 的可直接生成测试骨架;`实拍` 的留给人工/截图核验。
- **spec→设计稿**:`实拍` observable(出现/不出现/态)= 设计稿需画出的屏与状态。
- 这让「验收」同时是 code 的测试种子 + 设计稿的状态清单——一处定义,两边消费。
