## 一句话

出码、原型在本地跑起来之后,换一双没写过这代码的眼睛,装成第一次来的人,
照 SPEC 把"用户该干成的事"真上手走一遍,看干没干成、卡在哪,出一张能直接改的单子。

## 为什么接进 verify、不新建 skill

- "好不好看"已经有了 = verify 里现成的**品味层**(比兄弟页、你人眼终判)。不重起第二个视觉裁判。
- 真正缺的只有"好不好用"。它只在出码后顺手跑,不会脱开生成单独触发 → 当 verify 的一层就够,省一个 skill。
- 但它跟"出码的那个 agent"必须是**两副眼睛**:作者给自己判"好用"不可信(= 第二意见规矩)。所以这层由 **fresh-context 子代理**跑。

## verify 三层各兜什么

| 层 | 谁 | 兜什么 |
|---|---|---|
| 具象一致 | 机器(脚本) | label 位 / 星 / 控件宽 / 居中宽 —— 机械+契约+一致 |
| 品味层 | 你(设计侧人眼) | 长相像不像自家产品;审美终判 |
| **可用性走查(新)** | fresh 子代理 | 第一次来的人**干不干得成**、卡哪 |

> 注:"品味层"在 `verify-loop.md` 里其实只占两处,不是胖模块——
> ① 视觉证明要求:"与 canonical sibling 兄弟页做成对比较";
> ② 完成层级末档:`design-side release complete = 设计负责人验收 taste/fit`。
> 第三层与这两处**并列、不重叠**:它们判长相,它判干不干得成。

## 怎么跑

- **前置(老坑必查)**:先确认 localhost 那个端口**真正编译的是哪个仓库/worktree**,别假设是主仓库。
- **吃两样**:跑着的原型(子代理真去驱动:点 / 填 / 跳)+ 对应 SPEC(任务来源)。
- **任务从 SPEC 来**:照 SPEC 里"用户该做到的事"(用户目标 / 验收)去走。**装作没看过实现**——知道该达成什么,不知道该怎么点(只是别看*这个原型自己*的源码;别的现成页面可以看,当参照)。
- **锚参照怎么拿(不起环境、不用人手动截图)**:改法要锚到你们现成约定时——app 在跑、SPEC 点了兄弟页 → 子代理用**同一浏览器会话导航过去对照**;导不了(要登录 / 没数据等) → 凭已知约定锚 + **你终判兜底**。⚠️ 此条仅"真·app 内生成"成立(孤立原型没兄弟页可导),留下次真实生成坐实。
- **走查叙述固定四段式**(防空泛,不许只说"感觉怪"):
  > 我以为会… / 实际… / 卡在哪 / 严重度

## 走查必扫的隐性陷阱(广度别被头条挤掉)

> ⚠️ 实测教训:只盯 SPEC 主线,会漏掉**非阻断**的隐性坑(灰显可点、缺星号、静默)。定完头条后,这几类**逐项再扫一遍**再收尾:
> - **看着禁用的,真点一下**:灰显 / 低对比 / 像 disabled 的控件,实际点一次——灰显但可点 = 坑。
> - **逐字段查必填**:每个表单字段看有没有必填标识(星号);并**主动留空提交**,看是静默吞掉还是有明确报错。
> - **空态 / 反馈**:列表/区域空着有没有空态文案?提交后有没有 toast / 跳转 / 可见变化?
> - 确认头条(干没干成)后**别急着收尾**,把上面几类逐项过完再停。

## 不空泛的三个钉子

1. 任务**具体**(来自 SPEC),不是漫无目的乱看。
2. 每步用**固定四段式**叙述,不许只给"感觉"。
3. 每条改法必须**锚到现有产品**的某个 token / 组件 / 页,不许凭空的理想标准。

## 产出

- 一张清单,**落盘** `docs/specs/<需求>/`,对话只给 3–5 行摘要。
- 顶部:**能不能完成任务**的结论 + 最重的几条。
- 每条 = 位置 + 现象(我以为/实际) + 严重度 + 锚到现有产品的改法。
- 严重度三档:**卡死 / 绊脚 / 小瑕**(嫌多可砍两档)。
- 子代理**拿不准的单独标「待你拍」** —— 判断活,终判在人。

## 到哪为止 / gap 回流

- 出完清单就停。**不自动修复、不自动回喂生成**。
- 单子可在你**人工拍板后**回写 SPEC 决策日志(贴合确认门、不臆断、不自动改契约)。

## 明确不做(防"克苏鲁")

- 不打分 / 不评分卡。
- 不用脚本硬验"好不好用 / 好不好看"——核心是子代理推理 + 你拍。
- 不做 a11y / 性能 / code 质量审计(另外的活)。
- 不掺"好不好看"——那是品味层,要改去改品味层,不在这层。
- 不自动改代码。

> 「纯长相 vs 可用性」边界(双向,既防越界也防漏报):
> - 主色 RGB 差一点 / 圆角 / 间距 = **纯长相,不报**(归视觉门)。
> - 但视觉差异直接造成**动作语义分不清**(俩等权主按钮不知点哪个保存)= **算可用性,可报**,并连带提一句视觉。
