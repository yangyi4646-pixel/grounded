# 这两个 skill 在干嘛(一页总览)

> **初心:把粗需求「锚」到我们现有产品上——复用已有的页面 / 交互 / 字段 / 路由,堵住 AI 自己发明交互、发明设计。** 该你拍板处停下来问你。(= 从 1 不从 0)

## 流程

```
粗需求
  → 【prd-to-spec:锚需求,出 SPEC】
  → §0 冻结契约(复用哪些件 · 字段映射 · 路由/入口 · 冻结值)
  → 【spec-to-code:锚代码,出前端】
  → grounded 的 SPEC / 代码
```

## 它只干 4 件活(记这 4 件,不用记门)

| | 活 | 干什么 |
|---|---|---|
| ① | **锚到现有产品**(初心) | 复用现有页面 / 组件 / 字段 / 路由,**别发明** |
| ② | **补全** | PRD 没说的也补:空态/默认/**loading** · 错误/失败/**反馈** · 权限 · 数据一致性 · 谁读到 |
| ③ | **信息边界** | 不冗余、不把 A 的信息乱搬到 B 的页面 |
| ④ | **工作流 / 生命周期** | 对象在状态间流转时,**复用现有审批 / 状态机** |

## 它不干的

- **终判"好不好用 / 好不好看"的是你**(设计侧起预览、人眼看)。"好不好用"现由 fresh 子代理先走一遍出清单(verify 第 11 门)给你过;"好不好看"靠品味层。
- **不做工程交付质量**(types/lint/完成层级诚实之类) —— 那是 dev 侧的事。
- "出码后 AI 扮基础用户测可用性" = **已落地**为 spec-to-code verify 第 11 门(fresh 子代理走 SPEC 任务出可用性清单,你终判);A/B 实测 pass(召回 5/6 零方差,胜过无指引基线)。见 `spec-to-code/references/usability-walkthrough.md`。
